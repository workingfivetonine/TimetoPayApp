# Receipt Tracker

A mobile app for scanning receipts with AI, tracking prices over time, and building a smart shopping list.

## Run & Operate

- `pnpm --filter @workspace/api-server run dev` — run the API server (port 5000)
- `pnpm run typecheck` — full typecheck across all packages
- `pnpm run build` — typecheck + build all packages
- `pnpm --filter @workspace/api-spec run codegen` — regenerate API hooks and Zod schemas from the OpenAPI spec
- `pnpm --filter @workspace/scripts run generate-guide` — regenerate the offline how-to guide (MD + PDF) from `@workspace/guide-content`
- `pnpm --filter @workspace/scripts run generate-guide:check` — drift check (`guide-sync` validation): regenerates into a temp dir and fails if the committed guide MD / bundled PDF are stale
- `pnpm --filter @workspace/db run push` — push DB schema changes (dev only)
- `pnpm --filter @workspace/scripts run seed-stripe-price` — idempotently create the $5.99/mo Stripe product/price and print the `STRIPE_PRICE_ID` to set (checkout no longer applies a provider trial; the free trial is opt-in via `/billing/start-trial`)
- `pnpm --filter @workspace/scripts run seed-paypal-plan` — idempotently create the $5.99/mo PayPal product/plan and print the `PAYPAL_PLAN_ID` to set
- Required env: `DATABASE_URL` — Postgres connection string
- Required env: `AI_INTEGRATIONS_OPENAI_BASE_URL`, `AI_INTEGRATIONS_OPENAI_API_KEY` — Replit-provisioned OpenAI integration
- Optional env: `ADMIN_BOOTSTRAP_EMAILS` — comma-separated allowlist of verified emails permitted to bootstrap/recover the master admin role. **Unset ⇒ no user is ever auto-promoted (secure default).** Set this on a fresh deployment to the deployer-controlled email that should become admin.
- Optional env: `AI_GLOBAL_DAILY_MAX` — process-wide daily ceiling on AI-backed receipt-processing requests across ALL users (default `5000`). Bounds catastrophic Sybil/throwaway-account spend; raise/lower per deployment scale.
- Optional env: `CATALOG_CONTRIBUTOR_MIN_AGE_DAYS` — account-tenure gate for cross-user catalog contributors (default `7`). In the non-admin thresholded catalog view, a user's purchases only count toward the k-anonymity threshold once their account is at least this many days old; fresh accounts contribute price data but never unlock an entry. Removes the "on demand" sockpuppet probe (attacker must pre-age accounts). Set `0` to disable.

### Billing / subscription env (web-only paywall)

- Stripe: provided by the Replit **Stripe connector** (no manual secret key) — `stripeClient.ts` fetches the key via the connector proxy (`REPLIT_CONNECTORS_HOSTNAME` + `REPL_IDENTITY`/`WEB_REPL_RENEWAL`, injected automatically). Run the seed script then set `STRIPE_PRICE_ID`. The Stripe webhook is auto-managed by `stripe-replit-sync` (`findOrCreateManagedWebhook`), mounted at `/api/stripe/webhook` with the raw body parser BEFORE `express.json`.
- **Going live (test → live mode)**: the connector exposes separate development (test-mode) and production (live) connections; `stripeClient.ts` picks production only when `REPLIT_DEPLOYMENT=1`. To seed the **live** price you must override the connection from the dev workspace (where `REPLIT_DEPLOYMENT` is unset) via `STRIPE_CONNECTOR_ENVIRONMENT`. Steps: (1) activate a live Stripe account and connect its production connection in the Integrations tab; (2) run `STRIPE_CONNECTOR_ENVIRONMENT=production pnpm --filter @workspace/scripts run seed-stripe-price`; (3) set the printed `STRIPE_PRICE_ID` as the **production** secret; (4) republish. `STRIPE_CONNECTOR_ENVIRONMENT` accepts `production`|`development` only (kept in sync across both `stripeClient.ts` copies). The webhook is auto-managed per environment, so no manual webhook re-setup is needed.
- `STRIPE_PRICE_ID` — the recurring price id from `seed-stripe-price` (required for Stripe checkout).
- PayPal (manual env — no Replit connector exists): `PAYPAL_CLIENT_ID`, `PAYPAL_CLIENT_SECRET` (REST app credentials), `PAYPAL_PLAN_ID` (from `seed-paypal-plan`), `PAYPAL_WEBHOOK_ID` (the webhook id from the PayPal dashboard, used for server-side signature verification of `/api/webhooks/paypal`). Optional `PAYPAL_API_BASE` — **defaults to sandbox** (`https://api-m.sandbox.paypal.com`) in both `paypalClient.ts` and the seed script, so a LIVE deployment MUST set this to `https://api-m.paypal.com` explicitly.
- `PROMO_CODES` — comma-separated secret promo codes. A signed-in user redeems one via `POST /billing/redeem`, which sets the persistent `compAccess` flag → free full access. Treat codes as high-entropy secrets (constant-time compared server-side).
- `COMP_ACCESS_EMAILS` — comma-separated email allowlist always granted free access (checked live in `computeEntitlement`).
- If Stripe/PayPal env is unset the app boots normally (graceful no-op); checkout for the unconfigured provider returns an error, but trials, the promo/comp override, and the other provider keep working.

## Architecture decisions

- **Web deployment (real website + Expo Go mobile, same artifact)**: the published deployment serves BOTH the actual app in a browser AND the Expo Go mobile distribution. `scripts/build.js` runs `expo export --platform web` (SPA, `app.json` `web.output: "single"`) into `static-build/web/` IN ADDITION TO building the iOS/Android Expo Go bundles + manifests. `server/serve.js` routes by request: Expo Go (requests with `expo-platform: ios|android` header) get the platform manifest JSON; browsers get the real web app (`static-build/web/index.html` with SEO `<title>`/description/OG/Twitter/canonical/robots meta injected at serve time), with static assets resolved from `static-build/web` first then the timestamped Expo Go build, and a SPA fallback (extensionless paths → web index) so deep links like `/catalog` survive reload. `serve.js` also serves `/robots.txt` (Allow: /) and `/sitemap.xml` (Replit injects a `Disallow: /` robots.txt until the app serves its own). If the web build is missing, `serveWebApp` falls back to the old Expo Go QR landing page. Web build gets the same `EXPO_PUBLIC_*` env as the mobile build (domain, Clerk key, repl id). Camera falls back to image picker on web (already handled). **Changes only take effect after REPUBLISH.**
- **Public landing page**: `app/landing.tsx` is a public marketing homepage (RN, works web + native) shown to signed-out users. `app/_layout.tsx` `InitialLayout` redirects signed-out users on any non-public route to `/landing` (public routes = `(auth)` group + `landing`), and bounces signed-in users off `/landing`/`(auth)` to `/`. CTAs route to `(auth)/sign-up` / `sign-in`. SEO: `/` already renders the landing content for signed-out crawlers and is the single canonical/sitemap URL (`/landing` is the internal route, deliberately NOT in the sitemap and not separately canonicalized to avoid duplicate-content with `/`). Marketing copy/SEO meta strings live in `landing.tsx` + `server/serve.js` `buildSeoHead` — keep them roughly aligned. **Takes effect after REPUBLISH.**
- **Auth**: Replit-managed Clerk. Every user's data is fully private (stores/items/receipts carry a nullable `userId`, scoped per request by `requireAuth`). **Admin is bootstrapped from a trusted, deployer-controlled allowlist, NOT from being the first to sign up**: `ensureUser` (`middlewares/auth.ts`) promotes a brand-new user to master admin only when their **verified** Clerk primary email is in `ADMIN_BOOTSTRAP_EMAILS` (parsed by `lib/adminBootstrap.ts`) AND no admin exists yet (`NOT EXISTS` guard + the `users_single_admin_idx` partial unique index as the concurrency backstop). Public sign-up never confers admin by ordinal; with no allowlist configured the app stays admin-less by design. The app does NOT auto-absorb pre-existing ownerless data into the admin's personal views. Admin gets a read-only cross-user view (`/admin/*`, gated by `requireAdmin`). `GET /me` returns `{id, email, isAdmin, role}` so the client can show/hide admin UI. Expo client uses `@clerk/expo` with bearer tokens (works on web too); `setAuthTokenGetter` attaches the token to every generated API call. Sign-out + user switch clears the React Query cache so data never leaks across accounts. Clerk keys are auto-provisioned (`CLERK_PUBLISHABLE_KEY` → `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY` in dev script + build.js); never ask the user for them. Self-service password reset lives on the sign-in screen (Clerk Core reset-code flow) — Clerk has no admin backend "send reset email" API, so it's user-initiated.
- **User roles & management**: `usersTable.role` is `master_admin | family | general` (default `general`). `role` mirrors the `isAdmin` power flag (`master_admin` ⇔ `isAdmin`); `family`/`general` are labels only (no data sharing — every user's data stays private). Admin endpoints in `routes/admin.ts`: `PATCH /admin/users/:id/role` (assigning `master_admin` transfers admin in a row-locked txn — demote current, promote target, assert promotion or roll back), `POST /admin/users/merge` (reassign source's stores/items/receipts to target, dedup by normalized name, then delete source + Clerk user), `DELETE /admin/users/:id` (Clerk delete + DB cascade). The single-admin invariant is protected against concurrent races: delete/merge re-check + conditionally delete only when `isAdmin = false` (and merge row-locks the source `FOR UPDATE`), so you can never end up with zero admins.
- **Startup reconciliations** (`lib/bootstrap.ts`, run once per boot, all idempotent/self-disabling): `releaseLegacyAdminData` releases rows the admin owns whose `createdAt` predates the admin's account back to `userId = NULL` (undoes the old auto-claim behavior — they leave personal views but stay in the anonymized global catalog, which ignores `userId`); `reconcileAdminRole` backfills `role='master_admin'` for the existing admin; `ensureAdminExists` recovers admin ONLY by promoting a user whose email is in `ADMIN_BOOTSTRAP_EMAILS` (earliest match, **re-validated against Clerk's currently-verified primary email** — never the stale local `users.email`) when no admin exists — it never silently re-elects the earliest user (that would let an early public sign-up regain admin after an admin-less state). With no allowlist set it logs a warning and leaves the app admin-less.
- Contract-first: all endpoints defined in OpenAPI, hooks/validators auto-generated via Orval
- Numeric DB columns (price, total, deliveryFee) use `numeric`/string in Drizzle and are cast to `Number` in route responses
- Receipt AI parse uses OpenAI vision (`gpt-5.2`) via the Replit-managed `@workspace/integrations-openai-ai-server` lib — no API key needed in app code
- Stores and items are deduplicated by name (case-insensitive) during `parse-and-save`
- Each item has an emoji `icon` (nullable text): AI assigns one per line item at scan time; `iconForItemName(name)` keyword→emoji map (fallback 🛒) sets it on new items and lazily backfills existing items with null icon when re-encountered. Users can override it manually via an emoji picker on the item detail screen.
- Shopping list `isRecurring` flag = `purchaseCount >= 2` (tracked on itemsTable)
- Each item has a `category` (nullable text from a FIXED list in `lib/categories.ts`): AI assigns one per line item at scan time (mirrors the icon flow — set on new items, lazily backfilled on null); admin can correct it via the manage-catalog screen. Canonical catalog items also carry the category
- **Manage-catalog AI assists** (`routes/adminCatalog.ts`, admin-only): `POST /admin/catalog/items/suggest-categories` (body `{ids}`) classifies items via gpt-5.2 **batched** (`CATEGORY_BATCH=150`) so every requested id always gets a suggestion; each result validated against `FIXED_CATEGORIES`, falling back per-item to the `categoryForItemName` heuristic on omit/invalid/AI error. `POST /admin/catalog/items|stores/suggest-duplicates` (no body) ask gpt-5.2 to group same-real-world entities; AI returns nothing on failure (empty `[]`), ids are validated-against-existing + deduped across groups, single-shot capped at `AI_DUPLICATE_LIMIT=400` (one prompt must see all names to group them). `extractJson` tolerates code-fenced/prose-wrapped model output. Suggestions are advisory only — they feed the existing admin-reviewed merge/category UI, never auto-apply.
- **Catalog duplicate auto-suggestions** (`buildSuggestions` in `adminCatalog.ts`): union-find clustering over three signals — exact `looseKey`, token-sort key (word reordering), and Levenshtein similarity ≥0.85 (fuzzy pass gated to n≤800, name len≥4, length-ratio≥0.6 to bound cost). This is the non-AI baseline; the AI assist catches the garbled cases it still misses.
- **Shopping-list membership / dismissal**: an item is shown on the list if it has purchase history OR was explicitly added (`addedToListAt`), AND is not dismissed at/after its most recent event (`dismissedAt >= max(lastPurchased, ranOutAt, addedToListAt)`). `POST /items/:id/dismiss` sets `dismissedAt`; re-adding via catalog refreshes `addedToListAt` and clears `dismissedAt`. The browse endpoint's `inList` flag mirrors this exact logic — keep the two in sync (risk of drift)
- **Browse + add-to-list**: `GET /catalog/browse` (any authed user) returns catalog grouped by category with per-item `inList`; `POST /catalog/add-to-list` matches the user's existing item by ANY normalized alias of the canonical item (via `catalogItemAliasesTable`) to avoid duplicate user rows, then snapshots `globalPrice`/`globalStoreName` onto the item
- **Catalog privacy (k-anonymity)**: `computeGlobalPrices(opts?)` in `lib/catalog.ts` is shared by the trusted admin global view and the all-user browse/add-to-list paths. Called with NO opts it returns the full cross-user catalog UNCHANGED (admin only). The non-admin `/catalog/browse` AND `/catalog/add-to-list` pass `{ minDistinctUsers: CATALOG_MIN_CONTRIBUTORS, excludeUserId: userId }`: a per-store price is shown only when ≥`CATALOG_MIN_CONTRIBUTORS` (=3, covers the threat model's "singleton OR near-singleton") DISTINCT NON-NULL users bought it, the requester's own rows are excluded before counting (so it counts only OTHER users and never echoes the viewer's own data back), and an item with no surviving store is omitted entirely (no existence leak via `overallLatest*` or `stores[]`). Ownerless (`userId` NULL, anonymized legacy) rows contribute price data but never count toward the threshold or unlock an entry. **Sybil/sockpuppet hardening (`CATALOG_CONTRIBUTOR_MIN_AGE_DAYS`, default 7)**: raw distinct-userId counting is trivially gamed on a public self-service deployment — an attacker can spin up throwaway accounts + fabricated receipts to meet `CATALOG_MIN_CONTRIBUTORS` *on demand* and confirm a target's purchase (and approximate its month/price). So in the thresholded view only accounts older than the tenure gate count toward k-anonymity; fresh accounts behave like ownerless rows (feed price data, never unlock an entry). This removes the on-demand probe (an attacker must pre-provision and age sockpuppets, far costlier and not real-time). It is NOT full Sybil-proofing (a patient attacker can still age accounts); complete resistance would need identity attestation, deliberately out of scope. Admin view (no suppression) is exempt — it counts everyone. `/catalog/add-to-list` resolves the target id ONLY from the thresholded result and 404s on a suppressed/missing id (indistinguishable), closing the id-probing oracle that previously leaked hidden canonical item names. `overallLatest*` is selected by (purchasedAt desc, createdAt desc, catalogStoreId asc) so the admin no-suppress output is deterministic and matches the old "most recent row overall" behavior.
- **Recommended price/store** (shopping list + PDF): uses the user's OWN purchase history when available, else falls back to the snapshotted global price (`priceSource` = `history` | `global` | `none`); `recommendedPrice`/`recommendedStoreName`/lowest/avg/store are all nullable ("no price yet")
- Camera is only available natively (iOS/Android); web falls back to image picker
- **AI receipt-processing abuse controls** (`middlewares/aiRateLimit.ts`, `aiAbuseGuard(opts)`): every model-backed endpoint is wrapped because auth alone is NOT an abuse control on a public deployment. `/receipts/detect-bounds|parse|parse-and-save` use `imageGuard` (field `imageBase64`, ≤10MB chars); `/receipts/parse-pdf` uses the stricter `pdfGuard` (field `pdfBase64`, ≤15MB chars, lower limits). Each guard enforces, per authenticated user: a burst-window rate limit, a rolling 24h quota, and a concurrency cap; plus a process-wide concurrency cap and a shared process-wide daily budget (`AI_GLOBAL_DAILY_MAX`, default 5000) that caps total spend regardless of how many throwaway accounts an attacker controls. Oversized base64 bodies are rejected with 413 BEFORE any model/render work; over-limit requests get 429 + `Retry-After`. State is in-memory per process (single-instance deployment); concurrency is reserved on admit and released exactly once on response `finish`/`close`, with an unref'd periodic sweep to bound map growth.
- **PDF parse resource safety** (`/receipts/parse-pdf`): the 4-page cap is applied to the EXPENSIVE step, not just the model input. `pdfParse(buf, {max: PDF_MAX_PAGES})` bounds text extraction, and `pdftoppm` is invoked with `-f 1 -l PDF_MAX_PAGES` so only the first pages rasterize (a crafted high-page-count PDF can no longer force whole-document rendering), plus a wall-clock `timeout` + `SIGKILL` so a pathological page can't pin a worker. The downstream `slice(0, PDF_MAX_PAGES)` remains as defense-in-depth.
- **Offline guide stays in sync with the in-app guide**: the how-to guide text lives ONCE in `@workspace/guide-content` (`GUIDE_SECTIONS` / `GUIDE_ADMIN_SECTIONS`). `app/help.tsx` renders it (mapping `imageFile` → static `require()` in `GUIDE_IMAGES`), and `pnpm --filter @workspace/scripts run generate-guide` rebuilds `docs/guide/Receipt-Tracker-Guide.{md,pdf}` (PDF via `pdfkit`) and copies the PDF into `artifacts/receipt-tracker/assets/guide/`. After editing guide copy or swapping a screenshot in `assets/images/guide/`, rerun the generator so the bundled PDF doesn't drift. The `guide-sync` validation (`generate-guide:check`) regenerates into a temp dir and fails if the committed MD / bundled PDF are stale, so drift is caught before merge. The PDF is made byte-reproducible by pinning a fixed `CreationDate` in the generator. When adding a section, add it to `guide-content` AND add its screenshot key to `GUIDE_IMAGES` (Metro requires static `require()` literals)

- **Web-only subscription paywall (provider-agnostic, OPT-IN trial)**: signup grants NO subscription — a brand-new user resolves to status `none` / not entitled (freemium free features still work; premium surfaces are locked). The 30-day free trial is **opt-in and one-time**: the user explicitly taps "Start free trial" (account screen or paywall) → `POST /billing/start-trial`, which stamps `users.trialStartedAt` (timestamptz, nullable, **never cleared**) race-safely (`WHERE trial_started_at IS NULL`) only when `computeEntitlement(user).canStartTrial`. Alternatively they Subscribe ($5.99/mo, Stripe OR PayPal). **Stripe checkout no longer sets `trial_period_days`** — "Subscribe" is a paid sub so a user can't stack the app trial + a provider trial for double free time. Subscription state is stored provider-agnostically on `usersTable` (`subscriptionStatus`, `subscriptionProvider`, `subscriptionCurrentPeriodEnd`, plus `stripeCustomerId`/`stripeSubscriptionId`/`paypalSubscriptionId`). `lib/billing/entitlement.ts` `computeEntitlement(user)` is the single source of truth: admins, `compAccess` rows, and `COMP_ACCESS_EMAILS` matches are always entitled (status `comped`); otherwise it derives `trialing` (from the `trialStartedAt`+`TRIAL_DAYS` window, NOT `createdAt`) / `active` / `past_due` / `canceled` / `none` + `entitled`, and `canStartTrial` (true only when `!admin && !comp && !trialStartedAt && status∈{null,none}`). `formatCurrentUser` embeds the entitlement (including `canStartTrial`) in `GET /me`. **No data backfill**: any user mid-implicit-trial at rollout transitions to `none` and can reclaim the one-time trial with one tap (deliberate — avoids fragile migration/cutoff logic).
- **Freemium model (premium gate, 403)**: the app is FREEMIUM, not all-or-nothing. Free (signed-in but unpaid/lapsed) web users keep FULL access to their OWN data — manual stores/items/receipts CRUD, line items, shopping list, and basic spend analytics (calendar + weekly totals) are all UNGATED. Only PREMIUM surfaces are gated: (1) AI receipt scanning (`/receipts/detect-bounds|parse|parse-and-save|parse-pdf`), (2) the cross-user global price catalog (`/catalog/browse` + `/catalog/add-to-list`), and (3) the deeper per-item price-history analytics (`/analytics/items/:id/price-history` — the item-detail `/items/:id/history` route stays FREE). `middlewares/requireEntitlement.ts` exports `requirePremium` (a `RequestHandler`) applied INLINE per-route on exactly those endpoints; there is NO blanket `router.use` gate anymore (data routers in `routes/index.ts` are mounted free). A denied premium request gets **403** with body `{error:"premium_required", message: PREMIUM_REQUIRED_MESSAGE, entitlement}` where `PREMIUM_REQUIRED_MESSAGE` is exactly `"Subscribe for access to premium AI features"`. (Deliberately 403, NOT 402 — 402 historically meant "whole app locked".) Native clients are NEVER paywalled (app-store IAP policy) — `requirePremium` bypasses any request carrying `x-client-platform: ios|android` (set by the client via `setClientPlatform(Platform.OS)`); admins/trial/comp/active users also bypass via `computeEntitlement`. **Limitation (documented threat): the native bypass trusts a client header, so a web user could spoof `x-client-platform: ios` to dodge the gate.** Accepted v1 tradeoff (low-sensitivity data; per-platform token attestation is disproportionate). Revisit if abuse appears. NOTE (inline-middleware quirk): adding `requirePremium` inline on a `:id` route widens Express's inferred `req.params.id` type, so the price-history route coerces with `parseInt(String(req.params.id))`.
- **Webhook-driven sync**: Stripe events go through `stripe-replit-sync` (`lib/billing/stripeSync.ts` `processStripeWebhook`) at `/api/stripe/webhook` (raw body parser mounted BEFORE `express.json` in `app.ts`); `mapStripeStatus` normalizes to our status enum and reconciles the user row. PayPal webhooks hit `/api/webhooks/paypal` (public, but `verifyPaypalWebhook` does server-side signature verification via `PAYPAL_WEBHOOK_ID` before trusting anything); `mapPaypalStatus` normalizes. The PayPal approval return lands on `/paywall?paypal=success&subscription_id=…`; the client calls `POST /billing/paypal/finalize` which fetches the subscription from PayPal server-side (never trusting the client's "success") before activating.
- **Promo / comp override (secret free access)**: `POST /billing/redeem` validates a code against `PROMO_CODES` (constant-time hash compare, `lib/billing/promo.ts`) and sets the persistent `compAccess` boolean → permanent free full access. `COMP_ACCESS_EMAILS` is a live email allowlist (no redemption needed). Both flow through `computeEntitlement` so they bypass the paywall everywhere uniformly. Codes are treated as secrets (high-entropy, never logged).

## Product

- **List search & sort** (`components/ListControls.tsx`, generic `ListControls<K>` = search `TextInput` + "Sort" pills): every list screen has a search box and sort control. Controls render as a sibling ABOVE the list (never inside `ListHeaderComponent`/`SectionList` header) so the `TextInput` keeps focus on each keystroke. Filter/sort is client-side via `useMemo` over already-fetched data; when a non-empty query yields zero rows each screen shows a query-aware "No matching …" empty state (distinct from the true-empty state). Per screen: Receipts (search store/notes; sort Recent/Price/Store), Stores (name/address; A–Z/Delivery fee, null fees last), Shopping List (name/category; A–Z/Price/Category, shared helper applied to both Regulars + One-offs sections), Admin Global prices (name/store; A–Z/Price/Recent), Admin Manage catalog (canonicalName/category/member names; A–Z/Most used, query resets on items↔stores tab switch). Browse Catalog (`/catalog`) had its own search+sort already and is the styling reference.
- **Receipts tab**: list of all receipts, tap to see line items, edit item names/notes, delete items
- **Scan**: camera viewfinder with AI extraction — point at receipt, tap capture; also supports gallery upload
- **Stores tab**: add/edit stores with delivery fee and minimum order tracking; tap for cost-benefit analysis. Edit modal has a "Delete Store" button (confirmation warns it also removes that store's receipts/line items via DB cascade)
- **Item detail**: "Delete Item" button at bottom (confirmation warns it removes the item from shopping list, price history, and every receipt's line items via cascade); on success navigates back
- **Shopping List tab**: auto-populated from scanned receipts; split into Regulars (2+ purchases) and One-offs; shows lowest price and best store. Header download button exports a printable PDF (via `expo-print` + `expo-sharing`) grouped by store, each with Regulars/One-offs subsections, checkboxes per item, and ran-out items italicized/tagged
- **Analytics tab**: weekly spend bars with HIGH/LOW flags at ±1 std dev; per-item price history (lowest/avg/highest)
- **Paywall** (`app/paywall.tsx`, web only): NO LONGER a forced redirect — under the freemium model `_layout.tsx` `InitialLayout` does not bounce lapsed users here. Instead it's reached on demand when a free web user taps an upsell on a premium surface (scan, catalog, analytics Items tab) or the Account "Subscribe" button. Shows the $5.99/mo plan + features, "Pay with card" (Stripe) / "Pay with PayPal" buttons that open the returned checkout URL, a promo-code field (`POST /billing/redeem`), and sign-out. Handles the PayPal return (`?paypal=success&subscription_id`) by calling finalize then refreshing `/me`.
- **Premium upsell (freemium client)**: `hooks/usePremiumLock.ts` returns `locked` = web AND `/me` loaded AND `!entitlement.entitled` (native + still-loading + entitled ⇒ never locked). `components/PremiumUpsell.tsx` is the shared lock card (icon, title, subtitle, Subscribe → `/paywall`). Locked surfaces: `scan.tsx` swaps the Choose Photo/Upload PDF buttons for a "Subscribe to unlock" CTA (Enter Manually / Log Items stay free) and maps a 403 from the AI endpoints to an upgrade prompt; `catalog.tsx` renders the upsell + disables the `useBrowseCatalog` query (`enabled:!locked`) so no 403 fires; `analytics.tsx` shows the upsell on the Items tab only (Calendar/weekly spend stay free).
- **Account: Subscription** (`app/account.tsx`, web only): a vertical `subCard` shows the live entitlement status — default label is **"No subscription"** (never "Free trial"), trialing shows "Free trial · ends {date}", plus active/renews-date/past-due/comped. Actions: "Manage" (Stripe billing portal / PayPal management URL via `POST /billing/manage`) when a provider sub exists; "Start free trial" (only when `entitlement.canStartTrial`, calls `useStartFreeTrial`) + "Subscribe" → `/paywall` otherwise.
- **Browse Catalog** (`/catalog`): available to EVERY user (grid button in Shopping List header). Global price catalog grouped by category (most-recent price per canonical item, aggregated across all users without exposing who bought what — privacy-thresholded so only items/store prices ≥3 distinct OTHER shoppers have bought are shown; see "Catalog privacy" above). Each row has an add/check button to add the item to the user's shopping list. Items already active on the user's list show as checked; dismissed items show as addable again. Caption reads "Prices from items multiple shoppers have bought (we never show whose)."
- **Admin: Global prices** (`/admin/global`): admin-only cross-user catalog showing the most-recent price per canonical item (overall + per-store, lowest highlighted), aggregated across ALL users without exposing who bought what
- **Admin: Manage catalog** (`/admin/catalog`): admin-only tool to merge/rename/split spelling variants of item and store names into canonical entries. Surfaces auto-suggested merges (names that differ only by case/spaces/punctuation, plus word-reordering and fuzzy near-matches). Never mutates users' private rows — only the catalog mapping layer. Store entries also support an uploaded **logo**: stored as a base64 data URI on `catalogStoresTable.logo`, resized client-side (width 256 PNG via `expo-image-manipulator`) before upload, set/replaced/removed via `PATCH /admin/catalog/stores/:id` (which validates the data-URI MIME + ~1MB cap server-side). Two **AI assists** (gpt-5.2) in the list header: "Suggest categories" classifies every uncategorized item into the fixed category list — each shows as an inline AI pill the admin confirms (✓ → PATCH category) or rejects (✗ → keeps the manual "Set category" picker); "Find duplicates with AI" groups same-real-world entities (catches garbled/variant names exact/fuzzy matching misses) and feeds them into the same admin-reviewed merge flow (never auto-merges)

## Gotchas

- `expo-camera` must be v16.x (compatible with Expo SDK 54) — v56+ breaks with `createPermissionHook` error
- Drizzle `numeric` columns require string values on insert/update; cast to `Number` when returning JSON
- Run `pnpm run typecheck:libs` before `pnpm --filter @workspace/api-server run typecheck` to ensure lib declarations are built first
- `EXPO_PUBLIC_DOMAIN` env var is set by the workflow script — used in scan.tsx to construct the API URL
- The Stripe connector exposes the secret key as `settings.secret` (NOT `secret_key`) and requires the `X-Replit-Token` header + an `environment=development|production` query param (keyed off `REPLIT_DEPLOYMENT`) on the connector lookup — see `stripeClient.ts` (both the api-server and `scripts/src` copies).
- `stripe-replit-sync`'s `runMigrations()` resolves its SQL files relative to `__dirname` at runtime. Because the api-server is esbuild-bundled into `dist/index.mjs`, `__dirname` becomes `dist/`, so `build.mjs` copies the package's `migrations/` into `dist/migrations` — otherwise the migration step silently skips ("directory not found") and the `stripe.*` tables are never created (webhook setup then fails with `relation "stripe.accounts" does not exist`).
