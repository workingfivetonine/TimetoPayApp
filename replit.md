# Receipt Tracker

A mobile app for scanning receipts with AI, tracking prices over time, and building a smart shopping list.

## Run & Operate

- `pnpm --filter @workspace/api-server run dev` â€” run the API server (port 5000)
- `pnpm run typecheck` â€” full typecheck across all packages
- `pnpm run build` â€” typecheck + build all packages
- `pnpm --filter @workspace/api-spec run codegen` â€” regenerate API hooks and Zod schemas from the OpenAPI spec
- `pnpm --filter @workspace/scripts run generate-guide` â€” regenerate the offline how-to guide (MD + PDF) from `@workspace/guide-content`
- `pnpm --filter @workspace/scripts run generate-guide:check` â€” drift check (`guide-sync` validation): regenerates into a temp dir and fails if the committed guide MD / bundled PDF are stale
- `pnpm --filter @workspace/db run push` â€” push DB schema changes (dev only)
- `pnpm --filter @workspace/scripts run seed-stripe-price` â€” idempotently create the $5.99/mo Stripe product/price AND the annual $71.88/yr price + 20%-off "first year" coupon; prints `STRIPE_PRICE_ID`, `STRIPE_ANNUAL_PRICE_ID`, `STRIPE_ANNUAL_COUPON_ID` to set (checkout no longer applies a provider trial; the free trial is opt-in via `/billing/start-trial`)
- `pnpm --filter @workspace/scripts run seed-paypal-plan` â€” idempotently create the $5.99/mo PayPal product/plan and print the `PAYPAL_PLAN_ID` to set
- Required env: `DATABASE_URL` â€” Postgres connection string
- Required env: `AI_INTEGRATIONS_OPENAI_BASE_URL`, `AI_INTEGRATIONS_OPENAI_API_KEY` â€” Replit-provisioned OpenAI integration
- Optional env: `ADMIN_BOOTSTRAP_EMAILS` â€” comma-separated allowlist of verified emails permitted to bootstrap/recover the master admin role. **Unset â‡’ no user is ever auto-promoted (secure default).** Set this on a fresh deployment to the deployer-controlled email that should become admin.
- Optional env: `AI_GLOBAL_DAILY_MAX` â€” process-wide daily ceiling on AI-backed receipt-processing requests across ALL users (default `5000`). Bounds catastrophic Sybil/throwaway-account spend; raise/lower per deployment scale.
- Admin review-digest email (Gmail integration, optional): `ADMIN_DIGEST_INTERVAL_MS` (default `86400000` = daily), `ADMIN_DIGEST_MIN_GAP_MS` (default `43200000` = 12h; scheduled sends are skipped if the last send was more recent, so a restart loop can't spam), `ADMIN_DIGEST_INITIAL_DELAY_MS` (default `60000` = first tick 60s after boot). All no-ops if Gmail isn't connected or no admin email is on file.
- Optional env: `CATALOG_CONTRIBUTOR_MIN_AGE_DAYS` â€” account-tenure gate for cross-user catalog contributors (default `7`). **Currently INERT**: it only applies when a caller opts into the `minDistinctUsers` suppression in `computeGlobalPrices`, and the k-anonymity contributor threshold has been disabled by product decision (see the "Catalog privacy" bullet), so no caller passes it. Retained so the tenure-gated threshold can be re-enabled without code changes.

### Billing / subscription env (web-only paywall)

- Stripe: provided by the Replit **Stripe connector** (no manual secret key) â€” `stripeClient.ts` fetches the key via the connector proxy (`REPLIT_CONNECTORS_HOSTNAME` + `REPL_IDENTITY`/`WEB_REPL_RENEWAL`, injected automatically). Run the seed script then set `STRIPE_PRICE_ID`. The Stripe webhook is auto-managed by `stripe-replit-sync` (`findOrCreateManagedWebhook`), mounted at `/api/stripe/webhook` with the raw body parser BEFORE `express.json`.
- **Going live (test â†’ live mode)**: the connector exposes separate development (test-mode) and production (live) connections; `stripeClient.ts` picks production only when `REPLIT_DEPLOYMENT=1`. To seed the **live** price you must override the connection from the dev workspace (where `REPLIT_DEPLOYMENT` is unset) via `STRIPE_CONNECTOR_ENVIRONMENT`. Steps: (1) activate a live Stripe account and connect its production connection in the Integrations tab; (2) run `STRIPE_CONNECTOR_ENVIRONMENT=production pnpm --filter @workspace/scripts run seed-stripe-price`; (3) set the printed `STRIPE_PRICE_ID` as the **production** secret; (4) republish. `STRIPE_CONNECTOR_ENVIRONMENT` accepts `production`|`development` only (kept in sync across both `stripeClient.ts` copies). The webhook is auto-managed per environment, so no manual webhook re-setup is needed.
- `STRIPE_PRICE_ID` â€” the monthly recurring price id from `seed-stripe-price` (required for Stripe monthly checkout).
- `STRIPE_ANNUAL_PRICE_ID` â€” the annual ($71.88/yr) recurring price id from `seed-stripe-price` (required for the annual offer checkout; no auto-discovery fallback like the monthly price has). `STRIPE_ANNUAL_COUPON_ID` (optional) â€” the 20%-off "first year" coupon (`duration: once`) applied as a checkout `discounts` entry when `plan: "annual"`. Annual is Stripe-only (PayPal annual checkout returns 400); the seed script creates both and is idempotent. Same devâ†’live override story as `STRIPE_PRICE_ID` (`STRIPE_CONNECTOR_ENVIRONMENT=production`).
- PayPal (manual env â€” no Replit connector exists): `PAYPAL_CLIENT_ID`, `PAYPAL_CLIENT_SECRET` (REST app credentials), `PAYPAL_PLAN_ID` (from `seed-paypal-plan`), `PAYPAL_WEBHOOK_ID` (the webhook id from the PayPal dashboard, used for server-side signature verification of `/api/webhooks/paypal`). Optional `PAYPAL_API_BASE` â€” **defaults to sandbox** (`https://api-m.sandbox.paypal.com`) in both `paypalClient.ts` and the seed script, so a LIVE deployment MUST set this to `https://api-m.paypal.com` explicitly.
- `PROMO_CODES` â€” comma-separated secret promo codes. A signed-in user redeems one via `POST /billing/redeem`, which sets the persistent `compAccess` flag â†’ free full access. Treat codes as high-entropy secrets (constant-time compared server-side).
- `COMP_ACCESS_EMAILS` â€” comma-separated email allowlist always granted free access (checked live in `computeEntitlement`).
- If Stripe/PayPal env is unset the app boots normally (graceful no-op); checkout for the unconfigured provider returns an error, but trials, the promo/comp override, and the other provider keep working.

## Architecture decisions

- **Web deployment (real website + Expo Go mobile, same artifact)**: the published deployment serves BOTH the actual app in a browser AND the Expo Go mobile distribution. `scripts/build.js` runs `expo export --platform web` (SPA, `app.json` `web.output: "single"`) into `static-build/web/` IN ADDITION TO building the iOS/Android Expo Go bundles + manifests. `server/serve.js` routes by request: Expo Go (requests with `expo-platform: ios|android` header) get the platform manifest JSON; browsers get the real web app (`static-build/web/index.html` with SEO `<title>`/description/OG/Twitter/canonical/robots meta injected at serve time), with static assets resolved from `static-build/web` first then the timestamped Expo Go build, and a SPA fallback (extensionless paths â†’ web index) so deep links like `/catalog` survive reload. `serve.js` also serves `/robots.txt` (Allow: /) and `/sitemap.xml` (Replit injects a `Disallow: /` robots.txt until the app serves its own). If the web build is missing, `serveWebApp` falls back to the old Expo Go QR landing page. Web build gets the same `EXPO_PUBLIC_*` env as the mobile build (domain, Clerk key, repl id). Camera falls back to image picker on web (already handled). **Changes only take effect after REPUBLISH.**
- **Public landing page**: `app/landing.tsx` is a public marketing homepage (RN, works web + native) shown to signed-out users. `app/_layout.tsx` `InitialLayout` redirects signed-out users on any non-public route to `/landing` (public routes = `(auth)` group + `landing`), and bounces signed-in users off `/landing`/`(auth)` to `/`. CTAs route to `(auth)/sign-up` / `sign-in`. SEO: `/` already renders the landing content for signed-out crawlers and is the single canonical/sitemap URL (`/landing` is the internal route, deliberately NOT in the sitemap and not separately canonicalized to avoid duplicate-content with `/`). Marketing copy/SEO meta strings live in `landing.tsx` + `server/serve.js` `buildSeoHead` â€” keep them roughly aligned. **Takes effect after REPUBLISH.**
- **Auth**: Replit-managed Clerk. Every user's data is fully private (stores/items/receipts carry a nullable `userId`, scoped per request by `requireAuth`). **Admin is bootstrapped from a trusted, deployer-controlled allowlist, NOT from being the first to sign up**: `ensureUser` (`middlewares/auth.ts`) promotes a brand-new user to master admin only when their **verified** Clerk primary email is in `ADMIN_BOOTSTRAP_EMAILS` (parsed by `lib/adminBootstrap.ts`) AND no admin exists yet (`NOT EXISTS` guard + the `users_single_admin_idx` partial unique index as the concurrency backstop). Public sign-up never confers admin by ordinal; with no allowlist configured the app stays admin-less by design. The app does NOT auto-absorb pre-existing ownerless data into the admin's personal views. Admin gets a read-only cross-user view (`/admin/*`, gated by `requireAdmin`). `GET /me` returns `{id, email, isAdmin, role}` so the client can show/hide admin UI. Expo client uses `@clerk/expo` with bearer tokens (works on web too); `setAuthTokenGetter` attaches the token to every generated API call. Sign-out + user switch clears the React Query cache so data never leaks across accounts. Clerk keys are auto-provisioned (`CLERK_PUBLISHABLE_KEY` â†’ `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY` in dev script + build.js); never ask the user for them. Self-service password reset lives on the sign-in screen (Clerk Core reset-code flow) â€” Clerk has no admin backend "send reset email" API, so it's user-initiated.
- **User roles & management**: `usersTable.role` is `master_admin | family | general` (default `general`). `role` mirrors the `isAdmin` power flag (`master_admin` â‡” `isAdmin`); `family`/`general` are labels only (no data sharing â€” every user's data stays private). Admin endpoints in `routes/admin.ts`: `PATCH /admin/users/:id/role` (assigning `master_admin` transfers admin in a row-locked txn â€” demote current, promote target, assert promotion or roll back), `POST /admin/users/merge` (reassign source's stores/items/receipts to target, dedup by normalized name, then delete source + Clerk user), `DELETE /admin/users/:id` (Clerk delete + DB cascade). The single-admin invariant is protected against concurrent races: delete/merge re-check + conditionally delete only when `isAdmin = false` (and merge row-locks the source `FOR UPDATE`), so you can never end up with zero admins.
- **Startup reconciliations** (`lib/bootstrap.ts`, run once per boot, all idempotent/self-disabling): `releaseLegacyAdminData` releases rows the admin owns whose `createdAt` predates the admin's account back to `userId = NULL` (undoes the old auto-claim behavior â€” they leave personal views but stay in the anonymized global catalog, which ignores `userId`); `reconcileAdminRole` backfills `role='master_admin'` for the existing admin; `ensureAdminExists` recovers admin ONLY by promoting a user whose email is in `ADMIN_BOOTSTRAP_EMAILS` (earliest match, **re-validated against Clerk's currently-verified primary email** â€” never the stale local `users.email`) when no admin exists â€” it never silently re-elects the earliest user (that would let an early public sign-up regain admin after an admin-less state). With no allowlist set it logs a warning and leaves the app admin-less.
- **Admin review-digest email** (`lib/adminDigest.ts` + `lib/adminDigestScheduler.ts`, delivery via the Replit **Gmail connector** `google-mail` â€” `lib/email/gmailClient.ts` `sendGmail`, fresh `getUncachableGmailClient()` per send so OAuth tokens refresh): the single `master_admin` is emailed a digest of NEW things to review â€” new `catalog_items`, new `catalog_stores`, and new `users` â€” since the last send. A singleton `admin_notification_state` row (`lib/db/src/schema/adminNotifications.ts`) holds `lastDigestSentAt` as the high-water cursor; the digest covers the half-open window `(cursor, now]` so there are no gaps or double-counts. The window upper bound `until` comes from DB `now()` (not app time) so it can't drift from rows' DB-stamped `createdAt`; it's read as text and coerced via `new Date()` (a bare `sql\`now()\`` returns a string that would otherwise break Drizzle's timestamp param mapping). `runAdminDigest({trigger})`: **scheduled** enforces `ADMIN_DIGEST_MIN_GAP_MS`, skips when nothing is new, and consumes the queue by ATOMICALLY CLAIMING the window before sending â€” a conditional `UPDATE ... WHERE last_digest_sent_at IS NOT DISTINCT FROM since` advances the cursor to `until` only if it still equals the value just read (so overlapping runners can't double-send; a losing claim returns `reason: "concurrent"`), and the claim is ROLLED BACK if the Gmail send throws so the unsent window is reported next time; **manual** (admin-only `POST /admin/review-digest/test`) is a test/preview that ALWAYS sends (even empty, as a wiring confirmation), ignores the min-gap, and does NOT claim/advance the cursor. An unref'd `setInterval` (`startAdminDigestScheduler`, started in `index.ts`) drives the cadence, with a process-local in-flight guard so two scheduled ticks never overlap within the process (the only case where claim+rollback could drop a window); the first scheduled tick after a fresh boot has a null cursor so it reports ALL existing data once, then advances. Recipient = the admin's `users.email`; send failures / missing Gmail / missing admin email are logged and fail safe (never crash the process). NOTE: in-process timer is single-instance and scale-to-zero may miss ticks â€” acceptable for a low-stakes digest (next tick reports everything since the cursor).
- Contract-first: all endpoints defined in OpenAPI, hooks/validators auto-generated via Orval
- Numeric DB columns (price, total, deliveryFee) use `numeric`/string in Drizzle and are cast to `Number` in route responses
- Receipt AI parse uses OpenAI vision (`gpt-5.2`) via the Replit-managed `@workspace/integrations-openai-ai-server` lib â€” no API key needed in app code
- Stores and items are deduplicated by name (case-insensitive) during `parse-and-save`
- Each item has an emoji `icon` (nullable text): AI assigns one per line item at scan time; `iconForItemName(name)` keywordâ†’emoji map (fallback ðŸ›’) sets it on new items and lazily backfills existing items with null icon when re-encountered. Users can override it manually via an emoji picker on the item detail screen.
- Shopping list `isRecurring` flag = `purchaseCount >= 2` (tracked on itemsTable)
- Each item has a `category` (nullable text from a FIXED list in `lib/categories.ts`): AI assigns one per line item at scan time (mirrors the icon flow â€” set on new items, lazily backfilled on null); admin can correct it via the manage-catalog screen. Canonical catalog items also carry the category
- **Manage-catalog AI assists** (`routes/adminCatalog.ts`, admin-only): `POST /admin/catalog/items/suggest-categories` (body `{ids}`) classifies items via gpt-5.2 **batched** (`CATEGORY_BATCH=150`) so every requested id always gets a suggestion; each result validated against `FIXED_CATEGORIES`, falling back per-item to the `categoryForItemName` heuristic on omit/invalid/AI error. `POST /admin/catalog/items|stores/suggest-duplicates` (no body) ask gpt-5.2 to group same-real-world entities; AI returns nothing on failure (empty `[]`), ids are validated-against-existing + deduped across groups, single-shot capped at `AI_DUPLICATE_LIMIT=400` (one prompt must see all names to group them). `extractJson` tolerates code-fenced/prose-wrapped model output. Suggestions are advisory only â€” they feed the existing admin-reviewed merge/category UI, never auto-apply.
- **Catalog duplicate auto-suggestions** (`buildSuggestions` in `adminCatalog.ts`): union-find clustering over three signals â€” exact `looseKey`, token-sort key (word reordering), and Levenshtein similarity â‰¥0.85 (fuzzy pass gated to nâ‰¤800, name lenâ‰¥4, length-ratioâ‰¥0.6 to bound cost). This is the non-AI baseline; the AI assist catches the garbled cases it still misses.
- **Shopping-list membership / dismissal**: an item is shown on the list if it has purchase history OR was explicitly added (`addedToListAt`), AND is not dismissed at/after its most recent event (`dismissedAt >= max(lastPurchased, ranOutAt, addedToListAt)`). `POST /items/:id/dismiss` sets `dismissedAt`; re-adding via catalog refreshes `addedToListAt` and clears `dismissedAt`. The browse endpoint's `inList` flag mirrors this exact logic â€” keep the two in sync (risk of drift)
- **Browse + add-to-list**: `GET /catalog/browse` (any authed user) returns catalog grouped by category with per-item `inList`; `POST /catalog/add-to-list` matches the user's existing item by ANY normalized alias of the canonical item (via `catalogItemAliasesTable`) to avoid duplicate user rows, then snapshots `globalPrice`/`globalStoreName` onto the item
- **Catalog privacy (contributor threshold DISABLED by product decision)**: `computeGlobalPrices(opts?)` in `lib/catalog.ts` is shared by the trusted admin global view and the all-user browse/add-to-list paths. The former k-anonymity contributor threshold (require â‰¥3 DISTINCT users before showing a per-store price) has been **removed by product decision** â€” rationale: the catalog only ever exposes non-identifying aggregates (canonical item name, store name, a price, and a **month-coarsened** date), never a user identity or a raw per-user row, so a single contributor is treated as non-sensitive. Consequently a unique/rare item bought by even ONE other in-region shopper is now visible (no "who" is ever shown). The privacy controls that REMAIN: (1) **region scoping** â€” `/catalog/browse` + `/catalog/add-to-list` pass `{ countryCode, stateCode }`, so only stores in the viewer's own country (and US state) appear, and a region-less requester gets an empty catalog; (2) **own-data exclusion** â€” they also pass `excludeUserId: userId`, so the viewer's own rows are dropped before aggregating and the catalog never just echoes their data back; (3) **month-coarsening** of dates in the browse output; (4) `/catalog/add-to-list` still resolves the target id ONLY from the region-scoped result and 404s on an out-of-region/missing id (indistinguishable), so id probing can't disclose cross-region activity. The generic `minDistinctUsers` suppression (and its tenure gate `CATALOG_CONTRIBUTOR_MIN_AGE_DAYS`) is left intact in `computeGlobalPrices` but **no caller passes it**, so it's currently inert and trivially re-enableable. Admin view still calls with no opts (full cross-user visibility). `overallLatest*` is selected by (purchasedAt desc, createdAt desc, catalogStoreId asc) so the output is deterministic.
- **Recommended price/store** (shopping list + PDF): uses the user's OWN purchase history when available, else falls back to the snapshotted global price (`priceSource` = `history` | `global` | `none`); `recommendedPrice`/`recommendedStoreName`/lowest/avg/store are all nullable ("no price yet")
- Camera is only available natively (iOS/Android); web falls back to image picker
- **AI receipt-processing abuse controls** (`middlewares/aiRateLimit.ts`, `aiAbuseGuard(opts)`): every model-backed endpoint is wrapped because auth alone is NOT an abuse control on a public deployment. `/receipts/detect-bounds|parse|parse-and-save` use `imageGuard` (field `imageBase64`, â‰¤10MB chars); `/receipts/parse-pdf` uses the stricter `pdfGuard` (field `pdfBase64`, â‰¤15MB chars, lower limits). Each guard enforces, per authenticated user: a burst-window rate limit, a rolling 24h quota, and a concurrency cap; plus a process-wide concurrency cap and a shared process-wide daily budget (`AI_GLOBAL_DAILY_MAX`, default 5000) that caps total spend regardless of how many throwaway accounts an attacker controls. Oversized base64 bodies are rejected with 413 BEFORE any model/render work; over-limit requests get 429 + `Retry-After`. State is in-memory per process (single-instance deployment); concurrency is reserved on admit and released exactly once on response `finish`/`close`, with an unref'd periodic sweep to bound map growth.
- **PDF parse resource safety** (`/receipts/parse-pdf`): the 4-page cap is applied to the EXPENSIVE step, not just the model input. `pdfParse(buf, {max: PDF_MAX_PAGES})` bounds text extraction, and `pdftoppm` is invoked with `-f 1 -l PDF_MAX_PAGES` so only the first pages rasterize (a crafted high-page-count PDF can no longer force whole-document rendering), plus a wall-clock `timeout` + `SIGKILL` so a pathological page can't pin a worker. The downstream `slice(0, PDF_MAX_PAGES)` remains as defense-in-depth.
- **Offline guide stays in sync with the in-app guide**: the how-to guide text lives ONCE in `@workspace/guide-content` (`GUIDE_SECTIONS` / `GUIDE_ADMIN_SECTIONS`). `app/help.tsx` renders it (mapping `imageFile` â†’ static `require()` in `GUIDE_IMAGES`), and `pnpm --filter @workspace/scripts run generate-guide` rebuilds `docs/guide/Receipt-Tracker-Guide.{md,pdf}` (PDF via `pdfkit`) and copies the PDF into `artifacts/receipt-tracker/assets/guide/`. After editing guide copy or swapping a screenshot in `assets/images/guide/`, rerun the generator so the bundled PDF doesn't drift. The `guide-sync` validation (`generate-guide:check`) regenerates into a temp dir and fails if the committed MD / bundled PDF are stale, so drift is caught before merge. The PDF is made byte-reproducible by pinning a fixed `CreationDate` in the generator. When adding a section, add it to `guide-content` AND add its screenshot key to `GUIDE_IMAGES` (Metro requires static `require()` literals)

- **Web-only subscription paywall (provider-agnostic, OPT-IN trial)**: signup grants NO subscription â€” a brand-new user resolves to status `none` / not entitled (freemium free features still work; premium surfaces are locked). The 30-day free trial is **opt-in and one-time**: the user explicitly taps "Start free trial" (account screen or paywall) â†’ `POST /billing/start-trial`, which stamps `users.trialStartedAt` (timestamptz, nullable, **never cleared**) race-safely (`WHERE trial_started_at IS NULL`) only when `computeEntitlement(user).canStartTrial`. Alternatively they Subscribe ($5.99/mo, Stripe OR PayPal). **Stripe checkout no longer sets `trial_period_days`** â€” "Subscribe" is a paid sub so a user can't stack the app trial + a provider trial for double free time. Subscription state is stored provider-agnostically on `usersTable` (`subscriptionStatus`, `subscriptionProvider`, `subscriptionCurrentPeriodEnd`, plus `stripeCustomerId`/`stripeSubscriptionId`/`paypalSubscriptionId`). `lib/billing/entitlement.ts` `computeEntitlement(user)` is the single source of truth: admins, `compAccess` rows, and `COMP_ACCESS_EMAILS` matches are always entitled (status `comped`); otherwise it derives `trialing` (from the `trialStartedAt`+`TRIAL_DAYS` window, NOT `createdAt`) / `active` / `past_due` / `canceled` / `none` + `entitled`, and `canStartTrial` (true only when `!admin && !comp && !trialStartedAt && statusâˆˆ{null,none}`). `formatCurrentUser` embeds the entitlement (including `canStartTrial`) in `GET /me`. **No data backfill**: any user mid-implicit-trial at rollout transitions to `none` and can reclaim the one-time trial with one tap (deliberate â€” avoids fragile migration/cutoff logic).
- **Freemium model (premium gate, 403)**: the app is FREEMIUM, not all-or-nothing. Free (signed-in but unpaid/lapsed) web users keep FULL access to their OWN data â€” manual stores/items/receipts CRUD, line items, shopping list, and basic spend analytics (calendar + weekly totals) are all UNGATED. Only PREMIUM surfaces are gated: (1) AI receipt scanning (`/receipts/detect-bounds|parse|parse-and-save|parse-pdf`), (2) the cross-user global price catalog (`/catalog/browse` + `/catalog/add-to-list`), and (3) the deeper per-item price-history analytics (`/analytics/items/:id/price-history` â€” the item-detail `/items/:id/history` route stays FREE). `middlewares/requireEntitlement.ts` exports `requirePremium` (a `RequestHandler`) applied INLINE per-route on exactly those endpoints; there is NO blanket `router.use` gate anymore (data routers in `routes/index.ts` are mounted free). A denied premium request gets **403** with body `{error:"premium_required", message: PREMIUM_REQUIRED_MESSAGE, entitlement}` where `PREMIUM_REQUIRED_MESSAGE` is exactly `"Subscribe for access to premium AI features"`. (Deliberately 403, NOT 402 â€” 402 historically meant "whole app locked".) Native clients are NEVER paywalled (app-store IAP policy) â€” `requirePremium` bypasses any request carrying `x-client-platform: ios|android` (set by the client via `setClientPlatform(Platform.OS)`); admins/trial/comp/active users also bypass via `computeEntitlement`. **Limitation (documented threat): the native bypass trusts a client header, so a web user could spoof `x-client-platform: ios` to dodge the gate.** Accepted v1 tradeoff (low-sensitivity data; per-platform token attestation is disproportionate). Revisit if abuse appears. NOTE (inline-middleware quirk): adding `requirePremium` inline on a `:id` route widens Express's inferred `req.params.id` type, so the price-history route coerces with `parseInt(String(req.params.id))`.
- **Webhook-driven sync**: Stripe events go through `stripe-replit-sync` (`lib/billing/stripeSync.ts` `processStripeWebhook`) at `/api/stripe/webhook` (raw body parser mounted BEFORE `express.json` in `app.ts`); `mapStripeStatus` normalizes to our status enum and reconciles the user row. PayPal webhooks hit `/api/webhooks/paypal` (public, but `verifyPaypalWebhook` does server-side signature verification via `PAYPAL_WEBHOOK_ID` before trusting anything); `mapPaypalStatus` normalizes. The PayPal approval return lands on `/paywall?paypal=success&subscription_id=â€¦`; the client calls `POST /billing/paypal/finalize` which fetches the subscription from PayPal server-side (never trusting the client's "success") before activating.
- **Promo / comp override (secret free access)**: `POST /billing/redeem` validates a code against `PROMO_CODES` (constant-time hash compare, `lib/billing/promo.ts`) and sets the persistent `compAccess` boolean â†’ permanent free full access. `COMP_ACCESS_EMAILS` is a live email allowlist (no redemption needed). Both flow through `computeEntitlement` so they bypass the paywall everywhere uniformly. Codes are treated as secrets (high-entropy, never logged).

## Gotchas

- `expo-camera` must be v16.x (compatible with Expo SDK 54) â€” v56+ breaks with `createPermissionHook` error
- Drizzle `numeric` columns require string values on insert/update; cast to `Number` when returning JSON
- Run `pnpm run typecheck:libs` before `pnpm --filter @workspace/api-server run typecheck` to ensure lib declarations are built first
- `EXPO_PUBLIC_DOMAIN` env var is set by the workflow script â€” used in scan.tsx to construct the API URL
- The Stripe connector exposes the secret key as `settings.secret` (NOT `secret_key`) and requires the `X-Replit-Token` header + an `environment=development|production` query param (keyed off `REPLIT_DEPLOYMENT`) on the connector lookup â€” see `stripeClient.ts` (both the api-server and `scripts/src` copies).
- `stripe-replit-sync`'s `runMigrations()` resolves its SQL files relative to `__dirname` at runtime. Because the api-server is esbuild-bundled into `dist/index.mjs`, `__dirname` becomes `dist/`, so `build.mjs` copies the package's `migrations/` into `dist/migrations` â€” otherwise the migration step silently skips ("directory not found") and the `stripe.*` tables are never created (webhook setup then fails with `relation "stripe.accounts" does not exist`).
